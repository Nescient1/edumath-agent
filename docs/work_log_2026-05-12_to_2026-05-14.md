# EduMath Agent 工作日志：2026.05.12 - 2026.05.14

## 一、阶段目标

本阶段围绕 `edumath-agent` 的第一版可用闭环展开，重点不是做一个“大而全”的 AI 教育平台，而是把高中数学“函数与导数”专题的核心流程跑通：

```text
错题输入
-> 知识点识别
-> 错因诊断
-> RAG 检索讲义、题卡、错因规则
-> 个性化讲解
-> 相似题推荐
-> 学生作答
-> 自动批改
-> 更新学生画像
```

三天内主要完成了 RAG 数据库、OCR 输入、数据处理流水线、RAG Router、Embedding 策略、MiMoAI 接入、DOCX 提取优化和 PostgreSQL 数据库设计等工作。

### 2. 梳理 RAG 与大模型的关系

核心认知：

- RAG 知识库不是用来替代大模型，而是给大模型提供可控、可信、可追溯的外部材料。
- 大模型负责推理、表达和个性化讲解。
- 知识库负责教材体系、题库、讲义、错因规则、学生画像等系统记忆。

形成的策略：

```text
知识库优先
大模型兜底
资料类问题强调来源
数学求解类问题允许模型推理
推荐练习必须来自结构化题库
```

遇到的问题：

- 如果用户问“推荐三道题”，但向量库只检索到讲义，不能把讲义当作题目推荐。
- 单一 RAG 检索无法区分“讲概念”“解题”“推荐题”“查资料”“错因诊断”等意图。

解决方法：

- 设计 RAG Router，根据用户意图决定检索哪些内容类型。
- 推荐题必须来自 `questions.json` 或题目卡片，不允许从讲义生成伪题目。

### 7. 优化 DOCX 文本提取

遇到的问题：

`python-docx` 对数学公式支持较弱，很多公式识别不出来。

原因分析：

- 一部分公式是 Word OMML 数学对象。
- 另一部分公式是 WMF/EMF/OLE 图片对象。
- `python-docx` 只能较好读取普通段落文本，对公式和嵌入图片支持有限。

完成内容：

- 优化 `backend/scripts/extract_text.py`。
- 直接解析 DOCX 内部 XML：
  - `word/document.xml`
  - OMML 数学节点
  - 表格结构
  - 标题结构
- 增加提取质量记录：
  - `formula_count`
  - `image_count`
  - `extraction_quality`
  - `warnings`

遇到的新问题：

- 抽查真实 DOCX 后发现很多公式并不是 OMML，而是 WMF 图片。
- 因此 XML 解析只能解决一部分公式问题，不能完全解决图片公式问题。

解决方法：

- 在 manifest 中标记 `docx contains image(s); image formulas may need OCR`。
- 后续对图片公式单独做 OCR。

## 五、阶段性问题总结

### 问题 1：RAG 是否会限制大模型？

结论：

不会，前提是设计成“知识库优先，大模型兜底”，而不是“只能根据知识库回答”。

解决方案：

- 资料类问题优先 RAG。
- 数学题求解允许 LLM 推理。
- 推荐题必须依赖本地题库。
- 知识库低命中时说明使用通用数学方法。

### 问题 2：推荐题为什么不能直接用讲义？

结论：

讲义是知识解释材料，不是可作答的题目对象。

解决方案：

- 推荐题必须来自 `questions.json` 或 `question_card`。
- 讲义只能作为方法解释或解题参考。

### 问题 3：OCR 为什么不能直接接诊断？

结论：

OCR 可能识别错误，尤其是数学公式和条件。如果直接诊断，会把错误文本传给后续模型，导致整条链路偏离。

解决方案：

- OCR 后先自动填入输入框。
- 用户确认和修改。
- 再调用 `/api/diagnose`。

### 问题 4：DOCX 提取为什么效果不好？

原因：

- Word 中公式可能是 OMML，也可能是图片。
- `python-docx` 对图片公式无能为力。
- PDF/Word 资料中版式复杂，题目和解析容易被打散。

解决方案：

- XML 解析增强 OMML 提取。
- 图片公式标记为需要 OCR。
- 后续引入公式 OCR 和人工质检。

### 问题 5：批量资料是否应该直接用大模型清洗？

结论：

不建议第一版全量调用，成本不可控。

解决方案：

- 先做规则清洗。
- 再用 LLM 对少量样本做知识点标注和质量评估。
- 通过 PostgreSQL 保存每一步处理结果，便于回溯。
